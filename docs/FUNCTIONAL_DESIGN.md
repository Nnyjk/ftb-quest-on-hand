# Plan Board — 功能设计思路

> 解释每一个设计决策背后的"为什么"
> 目标读者：开发者（你我）
> 版本：v2.0.0

---

## 1. 问题起源

### 1.1 玩家的日常行为观察

在玩带有 JEI 的 mod 包时，一个高频操作循环：

> 打开 JEI → 看到想要合成的物品 → 逐个把材料加入书签 → 关闭 JEI → 打开箱子 → 对照着书签里的物品列表 → 在箱子里一个一个找 → 手动搬到背包

这个循环在每次制作一个复杂物品时都会重复一次。如果一个任务的合成链很长（比如钻石剑需要先合成木棍，再合成钻石锭，再合成钻石剑），玩家可能要在箱子和 JEI 之间来回切换 5~10 次。

### 1.2 核心痛点

**不是"不知道配方是什么"，而是"配方和容器之间缺乏连接"**

JEI 告诉你：
- 这个物品需要哪些材料 ✓
- 这个物品去哪找（来源） ✓
- **但它不会帮你把材料从箱子里取出来** ✗

玩家的手在"对照书签找物品"这件事上花了大量无意义的重复劳动。

---

## 2. 解决思路

### 2.1 为什么是"计划板"？

"计划"（Plan）这个词比"清单"更准确：
- 清单只是记录
- 计划暗示了"执行"的意图

计划板存储的不是零散的物品列表，而是**完整的合成配方**——它知道你要做什么，以及做这个东西需要什么。

### 2.2 为什么从 JEI 书签同步入手？

**因为玩家已经在这样做了。**

在没有任何 mod 辅助的情况下，高级玩家已经习惯：
- 把需要的物品加入 JEI 书签
- 利用 JEI 的书签来"记笔记"

这意味着：
- **没有额外的操作负担**：玩家不需要学习新的工作流
- **渐进式采用**：可以继续用 JEI 书签，同时享受计划板增量价值

JEI 书签是一个已经被玩家教育过的"记录行为"，我们寄生在这个行为上。

### 2.3 为什么不是自动从 JEI 同步所有书签？

**因为不是所有书签都是"我要合成的"**

玩家书签里有：
- 想要合成的物品（✓ 应该同步）
- 物品原型参考（如建筑方块，只想看看，不一定想合成）（✗ 不应该自动同步）

**解决方案**：JEI 书签监听是"添加时同步"，而不是"批量同步已有书签"。玩家主动把物品加书签这个行为本身就暗示了意图。

## 3. 设计决策

### 3.1 为什么只同步主合成配方？

一个物品在 JEI 中可能有多个合成表（钻石锭可以用工作台合成，也可以用紧实唱片机回收）。

**只取主合成配方**的原因：
- JEI 显示的第一个配方就是游戏内最常用的
- 简化逻辑，不需要做配方选择 UI
- 绝大多数情况下用户想要的就是主配方

### 3.2 为什么提取到背包而不是"全部从容器提取"？

**提取到背包的理由**：
- Minecraft 的使用场景是"边做边拿"
- 玩家通常就在工作台附近，背包是最自然的目的地
- 减少了额外的"从箱子拿回家"步骤

**为什么不直接进工作台网格？**
- 工作台网格是临时区域，物品用途未定义
- 背包才是真正的"已收入"状态

### 3.3 为什么背包满了就停止？

**为什么不强制塞进背包（覆盖已有物品）？**
- Minecraft 背包满了时，物品会掉在地上
- 掉在地上的物品会引来其他玩家的拾取（多人服务器）
- 或者在 PvP 中被其他玩家抢走

**为什么不提取到最近的空箱子？**
- 这需要定位"最近的箱子"，逻辑复杂
- 玩家可能不想让物品被放到其他箱子
- 背包满是一个明确的信号，让玩家做决定

### 3.4 为什么不支持 NBT 物品（暂定）？

NBT 物品（工具、防具、带附魔的物品、带组件的物品）的问题：
- 去重逻辑变得复杂：不能只看 registryName，还要比较 NBT
- 提取数量不明确：一把耐久度 50 的钻石剑和耐久度 100 的钻石剑算同一个物品吗？
- 这会大幅增加第一版的开发复杂度

**策略**：先做纯净物品的 MVP，NBT 支持作为正式迭代后的功能。

### 3.5 为什么配方容量需要配置？

**硬上限的理由**：
- 无上限的配方列表会导致：
  - NBT 体积膨胀
  - 提取时遍历成本高
  - UI 显示拥挤
- 9 个配方是一个心理舒适区（类似工作台的 9 格）

**可配置的理由**：
- 不同玩家有不同的需求
- 整合包可能有更复杂的配方链，需要更多槽位
- 配置化是最佳实践

## 4. 未采用的设计方案

### 4.1 为什么不用"扫描容器内所有配方"

**被否决的方案**：打开容器时，自动扫描容器内所有物品，识别哪些配方可以被满足。

**问题**：
- 计算量巨大（O(n²) 配方匹配）
- 与 JEI 的配方数据耦合深
- 玩家的行为模式是"我要做 X"，不是"看看容器能做什么"

**采用方案**：以计划板配方为驱动，只计算"缺什么"，不计算"能做什么"。

### 4.2 为什么不用语音/命令触发

**被否决的方案**：通过聊天命令 `/planboard extract` 触发提取。

**理由**：
- Minecraft 玩家不习惯在游戏内使用复杂命令
- Shift+右键的物理交互更自然
- 命令不适合初学者

### 4.3 为什么不做成方块形式

**被否决的方案**：将计划板做成可放置的方块，类似箱子。

**理由**：
- 增加了额外的方块 ID，可能与其他 mod 冲突
- 玩家需要记住"哪个是计划板箱子"
- 手持物品触发更直接，符合"我在这里处理"的直觉

## 5. 架构决策

### 5.1 核心层纯净的原因

与第一个方向一致：
- 核心提取逻辑可以被独立测试
- 不引入 Minecraft 类型的情况下可以做单元测试
- 三条兼容路线共享同一份核心逻辑

### 5.2 为什么三个兼容层分开？

与之前一致：
- NeoForge / Forge / Fabric 的 API 差异较大
- 分开可以针对每个平台最优实现
- 避免条件编译的复杂性

## 6. 未来扩展方向

以下功能不在当前 MVP 范围内，但设计时应考虑扩展性：

| 扩展功能 | 说明 |
|---------|------|
| NBT 物品支持 | 支持带组件/耐久的物品 |
| 多容器链式提取 | 从容器A提取后，将物品放入容器B |
| 配方优先级排序 | 按材料依赖关系自动排序（拓扑排序） |
| 统计面板 | 显示"已累计提取 X 个物品" |
| 导出/导入配方 | 将配方列表导出为 JSON |
