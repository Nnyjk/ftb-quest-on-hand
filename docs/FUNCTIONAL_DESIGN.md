# FTB Quest On Hand — 功能设计思路

> 解释每一个功能决策背后的"为什么"
> 目标读者：开发者（你我）

---

## 1. 核心设计哲学

### 1.1 问题起源

在玩带有 FTB Quests 的 mod 包时，有一个高频场景：

> 玩家打开任务书 → 找到一个想要完成的任务 → 需要收集 N 种材料 → 切出游戏打开 Wiki / JEI / NEI → 抄下材料清单 → 回到游戏开始找

这个过程打断了游戏节奏，尤其在需要一次性收集多种物品时，记忆负担重，来回切换成本高。

**核心问题**：任务目标物品（需要什么）和玩家手边（工作台附近）之间没有一条直接的通道。

### 1.2 解决思路

FTB Quest On Hand 的设计目标不是替代任务书，而是**缩短"任务 → 收藏 → 工作台"这条路径**：

```
传统路径（3步中断）：
  任务书 → [记忆/笔记] → 工作台

FTB Quest On Hand（1步直达）：
  任务书 → [侧边栏收藏] → 工作台
```

### 1.3 设计原则

| 原则 | 含义 |
|------|------|
| **不打扰** | 侧边栏默认隐藏，不影响原版工作台操作习惯 |
| **不替代** | 不做任务书，不做储物系统，只做"连接层" |
| **可撤回** | 所有操作（添加/移除）均可逆，没有强制绑定 |
| **可扩展** | 第三方 mod 可接入，不局限于 FTB Quests 和 JEI |

## 2. 侧边栏设计决策

### 2.1 为什么放右侧？为什么默认隐藏？

**放右侧**：
- 绝大多数玩家用右手操作鼠标，左侧空间留给主要注意力区域（工作台网格）
- 原版 Minecraft 工作台右侧是大片留白区，视觉上自然承接
- 不覆盖工作台 3×3 网格，不影响核心操作

**默认隐藏（折叠）**：
- 工作台是一个**高频、短时**操作界面（快速合成几组物品然后关闭）
- 如果侧边栏默认展开，每次打开工作台都要面对额外 UI 元素，长期使用会造成视觉疲劳
- 折叠设计尊重了用户的默认心智模型："需要时展开，不需要时消失"
- 展开按钮放在**右上角固定位置**，是受浏览器/IDE 侧边栏启发的通用交互模式，玩家无需学习

**为什么不做成单独的 GUI 界面？**
- 合成是高频操作，每多一次"开关界面"就多一层摩擦
- 折叠在工作台内部，合成流程无需切换上下文

### 2.2 为什么是 9 个槽？

9 是 3×3 工作台网格的列数，在视觉上形成呼应：
- 侧边栏每列 9 个槽，暗示它们是"工作台的延伸"
- 9 个物品足够装下一个中等任务的目标清单，又不至于太多导致需要滚动
- 如果未来需要更多槽，可加"下一页"或分类 Tab

**为什么不是无限滚动？**
- 无限滚动适合仓库/存储查看器
- 侧边栏是"收藏"而非"存储"，收藏的本质是精选，9个强制限制了总量，隐含了"只放真正需要的"这一设计意图

### 2.3 为什么支持拖拽？

拖拽是 Minecraft 原版就有的交互范式（背包内部拖拽、箱子拖拽）：
- 不需要额外的"添加按钮"和"移除按钮"
- 鼠标悬停在槽上时显示 tooltip，右键移除——这些操作在游戏中已经是肌肉记忆
- 拖入添加、拖出移除、左键点击填入工作台——三种操作覆盖了收藏的核心场景

## 3. FTB Quest 集成决策

### 3.1 为什么读取"目标物品"而非"奖励物品"？

**奖励物品**（Rewards）存在的问题：
- FTB Quests 的奖励在任务**完成后**才发放，此时玩家已经不需要"提醒收集"了
- 奖励的物品种类、数量都是任务设计者预设的，玩家没有选择权
- 奖励通常是一次性的，不会反复需要

**目标物品**（Objectives / 任务需求）才是真正的痛点：
- 玩家接了任务但还没完成，需要收集材料
- 收集行为本身是反复、持续的过程
- 玩家需要随时知道"我还需要什么"

**一个具体场景**：

```
任务：制作一把钻石剑
目标：3颗钻石 + 2根木棍
奖励：1个附魔台（假设）

传统体验：
  打开任务书 → 记住"钻石×3 木棍×2" → 关掉任务书 → 打开背包 → 开始找
  （找的过程中可能忘记数量，或者需要反复切回任务书核对）

FTB Quest On Hand 体验：
  打开任务书 → 点击"同步" → 钻石×3 木棍×2 自动出现在侧边栏
  → 侧边栏始终可见，随时对照收集进度
```

### 3.2 为什么是"同步"而不是"移动"？

任务目标是任务的**组成部分**，不属于玩家个人收藏。设计决策是：
- 侧边栏的是"参照"而非"提取"
- 物品不会被从任务书中移除，同步是**只读的引用**
- 玩家可以随时再次同步（以防任务目标中途变更）
- 可以同时将同一任务同步到侧边栏（快速收集）和 JEI 书签（快速查找来源）

### 3.3 为什么需要玩家主动触发同步，而不是自动同步？

**不自动同步的理由**：
- 不是所有已接取任务都需要当前收集
- 自动同步会污染收藏栏，玩家需要手动删除无用的
- 频繁的自动行为会增加认知负担
- 尊重玩家意愿：让玩家自己决定"我现在想做哪个任务"

**为什么需要选择界面（勾选列表）而不是一键全部同步**：
- 大型 mod 包可能同时接取 50+ 个任务，一键全部同步会填满 9 个槽
- 允许玩家只同步当前感兴趣的任务
- 每次只关注一个任务目标，符合"一次做一件事"的认知原则

## 4. JEI 书签同步决策

### 4.1 为什么要同步到 JEI 书签？

JEI 是玩家查找物品来源的**第一工具**：
- 几乎所有 mod 包玩家都会安装 JEI（或 REI）
- JEI 书签本身就用于"标记我想要的物品"
- 将任务目标添加到 JEI 书签 = 自动获得"这个物品去哪找"的能力

**JEI 书签和工作台侧边栏的功能区分**：

| 能力 | 侧边栏 | JEI 书签 |
|------|--------|---------|
| 随时可见 | ✅（展开时） | ❌ |
| 可直接拖入工作台 | ✅ | ❌ |
| 显示物品来源/合成表 | ❌ | ✅ |
| 物品数量不受限 | ❌（9个槽） | ✅ |
| 需要打开 JEI 才能操作 | ❌ | ✅ |

两者互为补充，不重复：

- **侧边栏**解决"合成时需要什么"
- **JEI 书签**解决"这个物品去哪找/怎么造"

### 4.2 为什么书签同步是独立功能？

JEI 书签的数量没有上限（不像侧边栏只有9格），
但需要打开 JEI 才能看到。

这意味着：
- 玩家可以将**所有当前任务的目标物品**都加入书签，然后在 JEI 里统一查看来源
- 侧边栏只放当前最紧急的 9 个（或其他上限）
- 两个功能针对不同场景，互不干扰

## 5. 外部 mod API 设计决策

### 5.1 为什么需要外部 API？

一个完整的功能向 mod 很难覆盖所有使用场景：
- AE2（应用能源）、精致存储（Refined Storage）等是 Minecraft 知名的存储系统
- 它们的用户有"从自己的存储里直接取物品到侧边栏"的需求
- 如果我们直接硬编码 AE2 集成，版本更新维护成本高，且无法覆盖所有存储 mod

**提供 API 的思路**：我们定义接口规范，让外部 mod 自己实现，核心层不感知细节。

### 5.2 为什么是"拉取"而非"监听"？

`fetchStoredItems()` 是主动拉取模式：
- 侧边栏打开时主动查询一次，不维护实时状态
- 外部存储可能有大量物品（AE2 终端可能存储上千种），实时同步成本高
- 玩家对实时性要求不高（不需要毫秒级更新）

### 5.3 为什么需要 `containsItem` 方法？

`containsItem` 的价值在于**筛选/高亮**：
- 当玩家有多个外部 mod 存储了同一个物品时，显示优先级
- 侧边栏可以标注"该物品在 AE2 中有库存"之类的提示
- 不是核心功能，但是 API 的合理延伸

## 6. 架构设计：核心 + 兼容层

### 6.1 为什么核心层必须纯净？

核心层（core）是这个项目的"宪法"：
- 它定义了所有业务逻辑的正确性基准
- 任何人都可以在不依赖 Minecraft / Forge / Fabric 的情况下测试核心逻辑

如果 core 层引入了任何 Minecraft 类型（如 `ItemStack`、`Item`），意味着：
- 单元测试需要启动完整 Minecraft 环境
- 三条兼容路线必须维护同一个"核心"，修改任何逻辑都要在三个地方改

因此 core 层的 `ItemStack` 使用 `String registryName` 而不是 `ResourceLocation`，
这是有意为之的**抽象隔离**。

### 6.2 为什么三条兼容层是独立模块而不是条件编译？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 条件编译（`@OnlyIf`）| 单仓库管理 | 构建配置复杂，测试困难 |
| **独立模块（当前方案）** | 完全隔离，互不影响，可独立测试 | 版本管理稍复杂 |
| 条件注解（Mixin）| 运行时替换 | 难以调试，行为难以预测 |

独立模块方案的额外好处：
- NeoForge / Forge / Fabric 的开发者可以只关心自己的模块
- 三条路的 API 表面一致，但内部实现可以针对平台最优
- CI 可以并行构建三个模块

### 6.3 为什么 NeoForge / Forge / Fabric 都支持？

FTB Quests 在三大主流 mod 加载器上都有用户：
- NeoForge（1.21+）：新兴平台，FTB Quests 已官方支持
- Forge（1.20.x）：历史存量最大，用户最多
- Fabric（1.20.x）：轻量 mod 包社区常用，FTB Quests 也有 Fabric 移植

不做某一条路意味着放弃该平台的全部用户，这不是功能向 mod 应该做的。

## 7. 数据持久化决策

### 7.1 为什么用 PlayerData 而不是全局配置文件？

每个玩家的收藏内容不同，必须按玩家分离存储。

PlayerData（`PlayerDataManager` / `PersistentCapabilities`）方案：
- 随玩家存档自动迁移（换存档不影响）
- 卸载 mod 后数据不污染存档
- Forge / NeoForge / Fabric 都有成熟实现

### 7.2 为什么是定时写入 + 退出时写入，而不是实时写入？

实时写入（每次收藏变更都写 NBT）的问题：
- 高频操作（如批量添加多个物品）会触发大量磁盘 IO
- Minecraft 主线程阻塞

定时（每60秒）+ 退出时写入的权衡：
- 最多丢失 60 秒的更改（可接受）
- 退出时必须写（防止玩家正常退出时数据丢失）
- 崩溃场景：最多丢 60 秒数据，崩溃本身已经是负面体验，轻微数据损失是可接受的退化

## 8. 未采用的设计方案及理由

| 被否决的方案 | 理由 |
|-------------|------|
| 侧边栏默认展开 | 干扰高频短时的合成操作，视觉疲劳 |
| 侧边栏无限槽 | 收藏本质是精选，强制上限有益设计 |
| 自动同步所有任务目标 | 污染收藏栏，增加删除负担 |
| 独立 GUI 界面 | 多一次界面切换，中断工作流 |
| 核心层引入 Minecraft 类型 | 破坏单元测试可行性，污染抽象 |
| 监听外部存储实时更新 | 性能成本高，实时性收益低 |
